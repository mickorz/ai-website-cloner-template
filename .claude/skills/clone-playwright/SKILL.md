# Clone Playwright

You are about to build a pixel-perfect Next.js clone from **pre-captured page data** generated by the `/open-playwright` skill.

Target: **$ARGUMENTS**

Unlike `/clone-website` (which requires live browser automation), this skill reads research documents and raw data already saved by `/open-playwright` in `docs/research/<domain>/`.

## Workflow Summary

```
/open-playwright <url>   -->   captures raw data + generates research docs
/clone-playwright <url>  -->   reads those docs + builds Next.js clone
```

**All work is done in the current session, sequentially. No sub-agents or worktrees.**

## Pre-Flight

1. Parse `$ARGUMENTS` as a URL. Extract the hostname (e.g., `gameui.ai`).
2. Check that research data exists:

```
docs/research/<hostname>/
  DESIGN_TOKENS.md
  COMPONENT_INVENTORY.md
  LAYOUT_ARCHITECTURE.md
  PAGE_TOPOLOGY.md
  components/*.spec.md
  <page-path>/
    screenshot.png
    computed-styles.json
    css-variables.json
    dom-structure.json
    page-links.json
    assets.json
    page.html
```

3. If research data is missing, tell the user to run `/open-playwright <url>` first, click the save button to capture the page, then re-run this skill.
4. Verify the base project builds: `npm run build`.

## Data Reading Order

Read these files IN ORDER. Each builds on the previous:

### 1. Design Foundation — `DESIGN_TOKENS.md`

Read this FIRST. It contains:
- **CSS Custom Properties** — color variables, spacing, radius
- **Extracted Colors** — all colors found on the page with usage counts
- **Typography** — font families, weights, sizes, line heights
- **Spacing Values** — padding, margin, gap patterns
- **Border Radius** — all radius values
- **Box Shadows** — all shadow definitions

**Action:** Update `src/app/globals.css` with the target site's design tokens:
- Map extracted colors to CSS custom properties
- Set `:root` variables for background, foreground, primary, etc.
- Add any custom properties not covered by shadcn tokens

Also read `css-variables.json` for the exact CSS variable values from the original site.

### 2. Component Inventory — `COMPONENT_INVENTORY.md`

Read this SECOND. It lists:
- **Page Sections** — all identified sections with tag, classes, children
- **Buttons & Links** — interactive elements
- **Headings** — h1-h4 with their classes and text
- **All Styled Elements** — every element with computed styles

**Action:** Map each section to a component name and target file:
- `<header>` + nav links → `src/components/Header.tsx`
- Hero section → `src/components/Hero.tsx`
- Feature sections → `src/components/Features.tsx`
- Workflow/Steps → `src/components/Workflow.tsx`
- CTA section → `src/components/CtaSection.tsx`
- `<footer>` → `src/components/Footer.tsx`

### 3. Page Layout — `PAGE_TOPOLOGY.md` + `LAYOUT_ARCHITECTURE.md`

Read these THIRD. They document:
- **Section order** — top-to-bottom page structure
- **Fixed/Sticky elements** — headers, floating buttons
- **DOM tree** — the full element hierarchy
- **z-index layers** — stacking context

**Action:** Plan the page assembly in `src/app/page.tsx`:
- Import all section components in order
- Set up sticky/fixed positioning
- Define z-index layers

### 4. Component Specs — `components/*.spec.md`

Read EACH spec file. Each contains:
- **DOM Structure** — the element hierarchy for that section
- **Computed Styles** — EXACT CSS values from `getComputedStyle()`
- **Text Content** — verbatim text from the live site
- **Assets** — images used in the section

**Action:** Use these as the blueprint for building each component.

### 5. Raw Data — `<page-path>/` files

Read as needed for additional detail:
- `computed-styles.json` — full CSS values for every matched element
- `dom-structure.json` — complete DOM tree
- `page-links.json` — all links for navigation
- `assets.json` — image/video/font inventory (src URLs, dimensions)
- `page.html` — reference HTML (use for text content and structure verification)
- `screenshot.png` — visual reference

## Build Phases

### Phase 1: Foundation

1. **Update fonts** in `src/app/layout.tsx`
2. **Update globals.css** with design tokens from DESIGN_TOKENS.md
3. **Download assets** — read `assets.json`, create a download script for all images:

```javascript
// scripts/download-assets.mjs
// Read assets.json from the research data
// Download each image to public/images/
```

4. **Extract SVG icons** from `svg-icons.json` and create `src/components/icons.tsx`
5. **Update next.config.ts** to add remote image domains from `assets.json`
6. Verify: `npm run build` passes

### Phase 2: Component Building (sequential, in current session)

For EACH component identified in the inventory, build it yourself:

1. **Read the spec file** `docs/research/<domain>/components/<name>.spec.md`
2. **Read the screenshot** for visual reference
3. **Read the relevant computed-styles.json** entries for exact CSS values
4. **Read the dom-structure.json** for that section's hierarchy
5. **Write the component file** directly:
   - Use the exact Tailwind classes from the DOM structure
   - Use verbatim text content from the spec
   - Use the exact CSS values from computed-styles.json
   - Use images/URLs from assets.json
   - Import shared utilities (cn(), icons) as needed
6. **Verify TypeScript** after each component: `npx tsc --noEmit`

Build components in this order:
1. Shared icons component (`src/components/icons.tsx`)
2. Layout components (Sidebar, Header)
3. Content sections (Hero, Features, Workflow, CTA, Footer, etc.)
4. Interactive components (forms, modals, etc.)

### Phase 3: Page Assembly

1. Wire everything in `src/app/page.tsx`
2. Import all section components in topology order
3. Connect real text content to component props
4. Verify: `npm run build` passes

### Phase 4: Visual QA

1. Run `npm run dev`
2. Compare clone with the screenshot in `docs/design-references/`
3. For each discrepancy, check the spec file and fix

## Important Notes

- Every CSS value in the spec files comes from `getComputedStyle()` — use these EXACT values, don't estimate
- Text content must be verbatim from the site, stored in spec files
- When a spec says `fontSize: 18px`, don't use `text-lg` — use the exact value
- Read the screenshot for visual verification but use the CSS values for precision
- If research data is incomplete for a section, note it but build what you can
- Build everything yourself in the current session — do NOT dispatch to sub-agents
